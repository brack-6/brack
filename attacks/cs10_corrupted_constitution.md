## Attack Summary

An agent co-authors a document and stores it at an external URL (e.g., a GitHub Gist). Later, an attacker with write access modifies the document to embed malicious instructions. On subsequent fetches, the agent trusts the document because it recognizes itself as a prior author â€” provenance becomes the attack vector. The injection arrives not via the prompt but via a URL the agent has a pre-established trust relationship with.

**Why this is distinct from standard indirect injection:**
- Standard indirect injection exploits documents the agent never touched
- CS10 exploits the *ownership/authorship heuristic* â€” "I wrote this, so it is safe"
- The trust is earned legitimately, then weaponized retroactively
- Detection must flag the combination of: external URL fetch + authorship claim + instruction content inside fetched document

## Why This Matters

Standard prompt-injection defenses inspect the prompt itself for adversarial signals. CS10 shows that trust can be pre-loaded into the agent's context through legitimate prior collaboration, then betrayed by a third party with write access to the shared artifact. The agent's own memory of authorship becomes the attack surface.

This is structurally similar to a supply chain attack: the agent trusts the artifact not because of its content at read time, but because of its *provenance*. Any agent that maintains a mental model of "documents I have written = safe to follow" is vulnerable.

**Detection heuristic:** Flag any fetched external content that contains instruction-bearing language, regardless of claimed authorship. Provenance does not transfer safety.
